# Regen PM AI 工作流优化方案

## 现状总结

你已经搭建了一个相当完整的 PM AI 工作流，覆盖四个核心阶段：

**需求 → HTML原型 → Figma设计 → Coding**

已有 8 个 Claude skill，核心管线是 `regen-material2proto → regen-proto2prd`，加上竞品分析、评审会、图片转原型等辅助能力。每个阶段都配了 AI 工具，并且有数据反馈和用户反馈的闭环设计。

---

## 一、管线级优化：打通断点

### 1.1 需求阶段的「输入层」还比较薄

**现状**：`regen-material2proto` 能处理会议记录、微信聊天、脑图，但输入基本依赖"用户手动粘贴文本"。

**建议**：

- **做一个 intake skill**：自动从文件夹/剪贴板读取多种格式（.docx 会议纪要、.pdf 客户邮件、.md 脑图导出、截图），预处理后统一喂给 material2proto。减少 Regen 的手动整理时间。
- **加入「需求优先级打分」**：在结构化摘要阶段，基于横纵分析法（你图里提到的），自动给识别到的功能点做 MoSCoW 或 RICE 打分，而不只是列清单。这样对客户的回复不是"行不行"，而是"先做哪些"。

### 1.2 原型 → Figma 的桥接是手动的

**现状**：图中标注了 "Web to Figma plugin"，但实际 skill 体系里没有覆盖这一步。

**建议**：

- 考虑做一个 `regen-proto2figma` skill，用 Figma MCP 或 vibma MCP 自动将 HTML 原型的结构（页面层级、组件、色值、间距）推送到 Figma。即使不能 100% 还原，至少生成一个「框架稿」让设计师在上面改，而不是从零开始。
- 或者反过来，如果设计师习惯从 Figma 起步，那在 Figma 里放一个「从原型导入」的 workflow，减少重复劳动。

### 1.3 Figma → Coding 缺少中间层

**现状**：图中标注 `regen-proto2prd` 用于这一步，但 PRD 到代码之间还有一个「技术方案」的鸿沟。

**建议**：

- 在 `regen-proto2prd` 的输出中加一个「技术实现建议」章节（API 接口清单、数据模型草案、组件拆分建议），让 vibe coding 的起点更高。
- 或者做一个 `regen-prd2techspec` skill，专门把 PRD 转成技术 spec，带 API endpoint 设计和数据库 schema 建议。

## 二、质量控制：从"能跑"到"跑得好"

### 2.1 评审会 skill 的使用时机可以前移

**现状**：`regen-review-board` 是一个 6 角色模拟评审，但看起来主要在 PRD 完成后使用。

**建议**：

- 在 `material2proto` 的第一步（结构化摘要）之后就跑一次轻量版评审——只关注「需求完整性」和「技术可行性」两个维度。发现缺失越早成本越低。
- 考虑做一个 `pm-quick-review` 轻量版，只做3个角色的快速检查（PM/Tech/UX），30秒出结果，适合在原型迭代过程中高频使用。

### 2.2 缺少用户视角的验证环节

**现状**：工作流有「数据反馈、用户反馈」的箭头，但没有对应的 skill。

**建议**：

- 做一个 `regen-usability-check` skill：给定原型 HTML，自动跑一遍认知走查（cognitive walkthrough），检查新手用户是否能完成核心任务流程，输出发现的障碍点。
- 在原型阶段就生成一份简单的「用户测试脚本」（5个核心任务+预期路径），让 Regen 拿去做非正式测试。

### 2.3 Evals 可以更系统化

**现状**：有 evals.json 但只有 2-3 个用例。

**建议**：

- 每个 skill 至少 5-8 个 eval 用例，覆盖：正常输入、边界输入（极少信息/信息冲突/多语言混杂）、错误处理。
- 加入回归测试：每次修改 skill 后自动跑全量 eval，避免改好一个场景破坏另一个。

## 三、效率提升：减少人工介入

### 3.1 材料预处理自动化

**现状**：每次都需要 Regen 手动准备材料并触发 skill。

**建议**：

- 做一个 watch-folder 模式的 scheduled task：监控特定文件夹，新文件落入后自动做初步分析（提取关键需求点、识别材料类型、标注信息密度），生成一个「材料就绪报告」，Regen 看一眼就知道够不够做原型。
- 利用 Claude 的 scheduled task 能力，可以做成每天早上自动汇总前一天收到的新材料。

### 3.2 原型迭代的版本管理

**现状**：原型迭代靠手动保存不同版本。

**建议**：

- 在 `material2proto` 里加入自动版本标记（v1, v2, v3），每次迭代自动生成 changelog（对比两版差异）。
- 利用 git 仓库（你已经有 .git），在 skill 里自动 commit 每个原型版本，commit message 带上修改原因。

### 3.3 上下文在 skill 之间的传递

**现状**：从 material2proto 到 proto2prd 需要手动告诉第二个 skill 用哪个文件。

**建议**：

- 建一个 `.regen/context.json`，记录当前项目的状态（当前阶段、原型路径、材料路径、确认过的决策点）。后续 skill 自动读取上下文，减少重复输入。
- 这相当于一个轻量的「项目状态机」，让整个管线更像一条自动流水线而不是手动接力。

## 四、Harness Engineering：怎么系统性提升每个环节

这是你图中底部提出的核心问题。我的理解是：如何让 AI 在每个环节做得更好、更稳定。

### 4.1 Prompt 层面

- **结构化 few-shot**：每个 skill 里放 2-3 个「黄金样本」（最佳输出案例），不只是 eval 用例，而是 skill 运行时会参考的范例。material2proto 可以附一个"优秀原型示例.html"。
- **负面示例**：明确告诉 AI "不要做什么"——比如 "不要在列表页加超过5个筛选条件"、"不要用弹窗替代详情页"。
- **思维链约束**：在关键决策点（如"是否需要这个页面"）要求 AI 先输出推理过程再给结论，减少随机性。

### 4.2 反馈闭环层面

- **每次使用后打分**：Regen 对输出打 1-5 分，附一句话评论，自动追加到 skill 的 learning_log.jsonl（space-image2proto 已经有这个机制，可以推广到所有 skill）。
- **定期回顾**：每月看一次 learning_log，找到共性问题，更新 skill prompt。这比"出问题了再改"高效得多。
- **A/B 对比**：对于核心 skill（material2proto），可以维护两个版本的 prompt，交替使用，对比哪个版本输出质量更高。

### 4.3 工具链层面

- **Design.md 标准化**：你列了 awesome-design-md、design-dna、impeccable 等资源。建议整合成一份 `design-system-spec.md` 放在项目根目录，所有 skill 统一引用，避免每个 skill 各自定义设计规范导致风格不一致。
- **MCP 连接器优化**：Figma MCP 的稳定性和速度是瓶颈。建议评估 vibma vs 原生 Figma MCP 的实际效果差异，选一个主力长期投入。

## 总结

你的工作流已经走在了大多数 PM 前面——有完整的管线、有辅助工具、有反馈闭环的意识。主要的提升空间在三个方向：

1. **连接断点**：skill 之间的上下文传递、原型到 Figma 的自动化
2. **质量闸门**：评审前移、用户视角验证、更丰富的 eval
3. **持续进化**：learning_log 全覆盖、定期回顾、prompt 版本管理
